# 大范围重构场景的追溯方案（候选生成 + 多信号打分）
目标：判断“新问题/修复是否已存在于历史分支（目标 ref）”，在文件移动、拆分合并函数、重命名、实现重写等情况下仍尽量稳定，并且在大仓库可运行。

## 总体框架
- 分两阶段：候选生成（高召回、低成本）→ 多信号打分/验证（高精度、可解释）
- 输出分层结论：FOUND / LIKELY_FOUND / NOT_FOUND / UNKNOWN，并附证据链（命中策略、文件/函数范围、得分、相关提交）

## L0：确定性（最高精度）
- Commit 链可达性：fix commit 是否为 target_ref 的祖先
- 适用：cherry-pick 保留提交关系、merge 导入等
- 结论：命中时可视为强结论

## L1：补丁/代码块等价（中高精度）
- 从修复提交提取修改证据：
  - hunk 上下文（前后几行）
  - 新增/删除的关键行（归一化后）
  - 受影响的文件路径集合（含 rename 线索）
- 在 target_ref 上搜索等价修改：
  - 优先在候选文件范围内搜索
  - 以“重叠率/包含率/上下文一致性”综合打分

## L2：符号/接口驱动的候选生成（重构主力）
- 从修复证据中抽取更稳定的线索：
  - 类/函数/方法名、导入路径、调用点模式
  - 关键常量字符串、错误码、日志模板
  - 外部 API 使用模式（例如调用了某个库的同一组接口）
- 在 target_ref 的代码快照上做快速检索，得到候选文件/候选函数集合

## L3：函数级稳健指纹（推荐的“规模化 AST”路线）
- 切分粒度：函数/方法（必要时 + 基本块）
- 指纹构建（示例组合）：
  - 词法 token 归一化（去空白/去字面量/变量名归一）+ winnowing 选指纹
  - 控制流骨架或 AST 节点类型序列 hash（弱化具体实现细节）
- 索引策略：
  - 对 target_ref 的快照建立一次索引（文件→函数→指纹）
  - 查询时只在候选集上做精细匹配（避免全历史遍历）

## L4：依赖/图信号（应对“实现重写但关系相近”）
- 近似 call graph / import 依赖 / 外部 API 集合 / 数据结构字段访问模式
- 用于再次缩小候选空间，或在 L3 得分接近时作为 tie-breaker

## 输入形态建议（提升重构下的稳定性）
- 只给 fix_commit：可用 L0/L1/L2/L3
- 更推荐：给 “问题窗口”：
  - bad_commit（问题存在）与 good_commit（问题修复后）
  - 以 bad..good 的差异提取更接近“bug 本质”的证据，明显提升大重构下召回

## 工程化要点
- 先召回后验证：不要在全仓库全历史做重度 AST 检索
- 所有结论都输出可解释证据：命中策略、相关文件/函数、得分构成、候选排名
- 阈值可配置，并用样本集做校准（Precision/Recall/F1 + 分策略分桶）
